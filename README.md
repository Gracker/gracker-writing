# Gracker Writing Skill

技术文章写作 skill。

## 安装

```bash
npx skills add Gracker/gracker-writing --yes --global
```

## 触发词

写文章、写公众号、写博客、帮我写篇稿、按我的风格写、技术长文写作

## 核心原则

**目标不是让读者觉得作者很懂，而是让读者真的看懂、能拿去用、知道边界。**

技术文章的核心是三件事：
1. **准确**：术语、版本、路径、代码、数据都能对上
2. **有用**：读者看完知道怎么观察、怎么判断、怎么落手
3. **易读**：不是把信息塞满，而是把复杂问题讲顺

## Sample

**修改前**（典型 AI 味，句句能顺翻回英文）：

> 真正决定渲染稳定性的，不是单点优化，而是整条链路的收口质量。系统的脆弱性比我们预期的更突出：context 不崩、state 可恢复、cache 命中率高，这三件事必须同时扛住。更锋利的做法是：把每一层的边界都 claim 硬，让问题自己在对的位置暴露出来。

**修改后**（骨架换回中文）：

> 渲染稳定性靠的是整条链路的收口质量，不是单点优化。系统比我们预期的更脆：上下文不乱、状态能恢复、缓存命中得上——这三件事同时做到才算稳。每一层把自己的边界定清楚，问题就会在对的位置暴露出来。

**对应 SKILL 规则**：

| 改动 | 规则 |
|------|------|
| `真正决定` → `靠的是` | 3.3.6 冗余确认副词 |
| `系统的脆弱性比预期的更突出：` → `系统比预期更脆：` | 3.3.10 抽象名词主语 + 形容词结论 |
| `更锋利的做法是：[内容]` → 删铺垫直接讲内容 | 3.3.9 形容词 + 冒号起手式 |
| `context / state / cache` → 上下文 / 状态 / 缓存 | 套路四 英文词混入（术语节） |
| `不崩 / 扛住 / 锋利 / claim 硬` → 不乱 / 做到 / 具体 / 定清楚 | 3.3.7 物理动作动词（翻译腔套路一） |

单独每一条都很"微小"，累加起来就是读者闻到的 AI 味。修改思路是：**先识别翻译腔骨架，再用中文本来的讲法重说一遍**——不是在原句上修修补补。
